Gako

Zero-knowledge secret management. The server that stores your secrets cannot read them — by design, not by promise.

In design — open source & self-hostable

Gako stores passwords, API keys, certificates, and notes encrypted end-to-end. Everything that touches plaintext happens on your device; the server holds only opaque ciphertext. A complete compromise of the server — its database, its backups, its administrators — reveals no secret content.

🔒 Truly zero-knowledge

No plaintext, no secret keys, and no means to derive them ever reach the server. What the server can and cannot see is explicitly enumerated, not hand-waved.

⚡ Instant revocation

Removing someone's access is immediate and server-side — no re-encryption, no waiting on other clients, no shared keys to rotate.

📋 Honest exposure accounting

Know exactly which secrets a departing user has seen, so you rotate what actually needs rotating — a concrete worklist instead of vague unease.

🧱 Boring, proven crypto

XChaCha20-Poly1305, X25519, Ed25519, and Argon2id, composed in well-trodden ways. No exotic cryptography in the core.

✍️ Tamper-evident

Every secret is signed by the device that wrote it and verified before anything is shown. A server that forges, alters, or rolls back data gets caught by your clients — not trusted.

🏠 Self-hosted in one binary

One static binary serves the API and the web app; it needs nothing but a data directory. Your secrets stay on your infrastructure — and they're unreadable even there.

📈 Scales both ways

From one user on one machine to a large organization with fine-grained, per-secret access control — same architecture, same security model.

🖥️ Clients everywhere

Web, CLI, desktop, mobile, and browser extension — one cryptographic core shared by all of them, audited once instead of five times.

Honest about what the server learns

Zero-knowledge applies to your secrets' content — not to everything. Gako enumerates the difference instead of hand-waving it.

The server stores and sees

  • Opaque ciphertext and its approximate, bucketed size
  • The access graph — which accounts may read which (encrypted) secrets
  • When clients connect, read, and write

The server can never read

  • Secret values, titles, URLs, usernames, notes — any content, current or historical
  • Your master password — login sends only a one-way derivative, useless for decrypting anything
  • Any key capable of decrypting anything it stores

One system, three jobs

A single architecture serves every usage profile without changing its security model.

Personal password manager Team & org secret sharing Secrets store for machines

Named after the Basque word for key — a language isolate, related to nothing else, just like a zero-knowledge server that knows nothing about your secrets.
